- **Описание:** Проигнорировано требование ТЗ использовать отдельные функции/методы для обработчиков событий. [Скриншот.](https://drive.google.com/uc?id=1EZGrU20JJec7pguzgf2zdcBjMfYQJ4fq)

### 20 - Нет сериализации данных

- **Описание:** С текущей логикой чем больше полей формы, тем больше будет кода. Следует проводить сериализацию данных в более автоматическом режиме, например, собирая все пары ключ-значение из FormData циклом. [Скриншот.](https://drive.google.com/uc?id=1vyxOtIqmFo1h0zNOO92H5Yvx7ZUHm7uq)
- **Исправлено:** [✓]
- **Комментарий:** Добавила сериализацию данных с помощью цикла

### 21 - Неправильное расположение входного модуля

- **Описание:** Хорошей практикой считается запуск модуля не в нем самом, а во входной точке приложения. [Скриншот.](https://drive.google.com/uc?id=1ESXPWWeyNkyfcElzpPu3Yt8-y8kKyBNJ)
- **Исправлено:** [✓]
- **Комментарий:** Вынесла запуск модуля в main.ts

### 22 - Неккоректный выбор валидатора

- **Описание:** Выбрана верная логика валидации, но она некорректно исполнена по части выбора валидатора. Не следует обращаться по классу поля, будет уместнее использование data-attr. Например, data-input-required="type", где "type" будет означать способ валидации данного поля. [Скриншот.](https://drive.google.com/uc?id=1xzoBtOIUeJh7RKroX3vjHe5UWvhxy0KQ)
- **Исправлено:** [✓]
- **Комментарий:** Переписала код с использованием data-input-required="type"

### 23 - Нарушение принципов SOLID

- **Описание:** Класс "ValidateForm" дословно означает "валидация формы". Почему тогда в нем присутствует логика по отправке на backend? Это нарушает принципы SOLID. Следует выделить еще один модуль, который отправляет данные.
- **Исправлено:** [✓]
- **Комментарий:** Вынесла ValidateForm в отдельный модуль
